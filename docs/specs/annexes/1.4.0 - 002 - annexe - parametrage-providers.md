| | |
|---|---|

## 1. Principes transverses

Le paramétrage des trois CLI répond aux exigences du CDC, dans cet ordre de priorité :

1. **Non-interactivité totale** (EXG-CAP) : aucune invocation ne doit pouvoir attendre une entrée humaine — mode print/exec obligatoire, approbations désactivées, permissions résolues par configuration en amont.
2. **Sortie machine** (EXG-CON) : JSON natif exigé partout où il existe ; validation par schéma quand la CLI le supporte nativement, sinon extraction par délimiteurs.
3. **Pinning de modèle** (EXG-ROL-01) : modèle imposé à chaque invocation ; les mécanismes de fallback automatique de modèle sont **désactivés** (un changement de modèle silencieux fausserait les statistiques EXG-SCO et la comparabilité des runs EXG-PRM).
4. **Reproductibilité et cloisonnement** (EXG-CTX, EXG-ROL-03) : désactivation des mécanismes de contexte implicite (mémoire automatique, hooks, plugins, MCP, fichiers d'instructions découverts) — tout le contexte vient du module `context/` d'AI-Forge. Sessions non persistées ou éphémères par défaut.
5. **Permissions par rôle** (EXG-SEC-01) : le moteur de politiques d'AI-Forge se traduit dans le mécanisme natif de chaque CLI (allow/deny lists, sandbox, modes lecture seule).
6. **Non-attribution** (INV-006, EXG-INV-03) : options natives quand elles existent, réécriture des commits sinon.
7. **Reprise** (EXG-QUO-02) : identifiants de session capturés quand disponibles, mais les prompts restant autoporteurs, la reprise nominale d'AI-Forge est une **ré-invocation fraîche** — la reprise de session native est une optimisation, pas une dépendance.

Chaque adaptateur (`src/providers/`) construit ses invocations exclusivement à partir de `providers.toml` ; aucun flag n'est codé en dur.

## 2. Claude Code (`claude`) — modèle imposé : Opus 4.8

### 2.1 Invocation type

```bash
# Rôle DEV (écriture autorisée dans le worktree)
claude -p "$(cat prompt_rendu.txt)" \
  --model opus-4.8 \
  --output-format json \
  --permission-mode dontAsk \
  --allowedTools "Bash(git *)" "Bash(pytest *)" "Bash(uv *)" "Edit" "Write" "Read" \
  --disallowedTools "WebFetch" "WebSearch" "mcp__*" \
  --strict-mcp-config \
  --bare \
  --no-session-persistence \
  --max-turns 80 \
  --settings '{"attribution": {"commit": "", "pr": ""}}' \
  --add-dir <worktree>            # exécuté avec cwd = worktree

# Rôle TESTER (exécution sans push)
#   idem avec --allowedTools "Bash(pytest *)" "Bash(ruff *)" "Bash(mypy *)" "Read"
#   et --disallowedTools "Edit" "Write" "Bash(git push *)"

# Rôle REVIEWER (lecture seule)
#   idem avec --tools "Read" (aucun outil d'écriture ni Bash)
#   + --json-schema pour le RoleVerdict (cf. 2.3)
```

### 2.2 Paramètres retenus et justification

| Paramètre | Valeur AI-Forge | Exigence servie |
|---|---|---|
| `-p` / `--print` | toujours | Mode headless non interactif. |
| `--output-format` | `json` (ou `stream-json` pour le suivi temps réel dans `forge status`) | EXG-CON ; le JSON de sortie inclut le résultat, le coût et l'identifiant de session. |
| `--json-schema <schema>` | schémas de `src/contracts/` (RoleVerdict…) | **Validation native du contrat de sortie** — Claude Code garantit un JSON conforme au schéma en fin de run ; réduit les relances EXG-CON-02. |
| `--model` | `opus-4.8` (configurable) | EXG-ROL-01. |
| `--fallback-model` | **jamais utilisé** ; setting `fallbackModel` absent | Interdiction du changement de modèle silencieux. |
| `--permission-mode` | `dontAsk` (DEV/TESTER) ; jamais `bypassPermissions` | Non-interactivité sans désactiver le moteur de permissions : les règles allow/deny restent évaluées, tout ce qui n'est pas autorisé est refusé au lieu de solliciter l'humain. |
| `--allowedTools` / `--disallowedTools` / `--tools` | générés depuis `policies.toml` par rôle | EXG-SEC-01 — c'est la traduction native de l'allowlist AI-Forge. |
| `--strict-mcp-config` (sans `--mcp-config`) | toujours | Aucun serveur MCP chargé : pas de canal de contexte ou d'action hors du contrôle d'AI-Forge. |
| `--bare` | toujours | Ignore CLAUDE.md, hooks, skills, plugins, mémoire automatique : le contexte vient exclusivement du prompt construit par `context/` (EXG-CTX-03, anti-injection EXG-SEC-06) ; démarrage plus rapide. |
| `--no-session-persistence` | défaut (levé si reprise native activée) | Cloisonnement EXG-ROL-03 ; rien ne fuit d'un rôle à l'autre via l'historique local. |
| `--session-id` / `--resume` / `--fork-session` | optionnel, si `providers.claude.native_resume = true` | Optimisation de reprise après épuisement (EXG-QUO-02) ; `--fork-session` évite de polluer la session d'origine. |
| `--max-turns` | plafond par rôle (défaut : 80 DEV, 40 TESTER, 20 REVIEWER) | Garde-fou budget (EXG-BUD) contre les boucles agentiques. |
| `--append-system-prompt-file` | fichier de rôle rendu par `prompts_registry/` | Injecte les règles de rôle en conservant les garde-fous par défaut de Claude Code (préférable à `--system-prompt` qui les supprimerait). |
| `--settings` (inline) | `{"attribution": {"commit": "", "pr": ""}}` | **INV-006 nativement** : supprime les mentions `Co-Authored-By`/`Generated with` des commits et PR. Le scan EXG-INV-03 reste actif en ceinture-bretelles. |
| `--exclude-dynamic-system-prompt-sections` | activé | Améliore la réutilisation du cache de prompt entre workers/machines — réduit la consommation. |
| `--add-dir` / cwd | worktree du BL uniquement | EXG-SEC-01 (écriture limitée au worktree). |

### 2.3 Vérifications et signaux

- **Health-check** : `claude -v` (version), `claude auth status` (code retour 0 = authentifié, JSON exploitable), invocation d'essai `-p` minimale vérifiant que `--model opus-4.8` est accepté.
- **Capacités déclarées** (`providers.toml`) : `non_interactive: true`, `json_output: true`, `json_schema_output: true`, `model_pinning: true`, `supports_no_attribution: true` (via setting `attribution`), `native_resume: true`.
- **Détection d'épuisement (EXG-QUO-01)** : sortie non-zéro + motifs de type limite d'usage/rate limit dans stderr ou le champ d'erreur JSON (fenêtres 5 h et hebdomadaires des abonnements). Les patterns exacts sont calibrés en v0.1.1 sur les messages réels et maintenus dans `providers.toml` (`exhausted_patterns = [...]`).
- **Points d'attention** : les paramètres `settings.json` utilisateur (`~/.claude/`) ne doivent pas interférer — `--setting-sources` peut restreindre les sources chargées si le poste a une config utilisateur riche ; `--bare` couvre déjà l'essentiel.

## 3. Codex CLI (`codex`) — modèle imposé : GPT-5.5

### 3.1 Invocation type

```bash
# Rôle DEV
codex exec - \
  --model gpt-5.5 \
  --profile ai-forge-dev \
  --sandbox workspace-write \
  --ask-for-approval never \
  --json \
  --output-last-message <run_dir>/last_message.txt \
  --output-schema <contracts>/role_verdict.schema.json \   # rôles à verdict
  --ephemeral \
  --cd <worktree> \
  --skip-git-repo-check=false \
  -c 'features.web_search="disabled"' \
  < prompt_rendu.txt

# Rôle TESTER : --profile ai-forge-tester, --sandbox workspace-write
#   (exécution de tests) + execpolicy interdisant git push
# Rôle REVIEWER : --profile ai-forge-reviewer, --sandbox read-only
```

### 3.2 Paramètres retenus et justification

| Paramètre | Valeur AI-Forge | Exigence servie |
|---|---|---|
| `codex exec` (`codex e`) | toujours | Mode non interactif scripté ; prompt lisible depuis stdin (`-`). |
| `--json` | toujours | Événements JSONL en streaming — alimente l'event log et `forge status`. |
| `--output-schema <path>` | schémas de `src/contracts/` | **Validation native du contrat** contre un JSON Schema. |
| `--output-last-message <path>` | fichier du run | Capture déterministe du message final, indépendante du parsing du flux. |
| `--model` / `-m` | `gpt-5.5` (configurable) | EXG-ROL-01. |
| `--profile` | un profil par rôle (`ai-forge-dev`, `-tester`, `-reviewer`) dans `~/.codex/config.toml` | Regroupe modèle, sandbox et options par rôle ; les profils sont déployés/vérifiés par `forge doctor`. |
| `--sandbox` | `workspace-write` (DEV/TESTER), `read-only` (REVIEWER) | **Sandboxing natif OS** (Seatbelt/Landlock/Windows) : écriture limitée au workspace = le worktree (EXG-SEC-01, complète EXG-SEC-04 sans conteneur). Jamais `danger-full-access`, jamais `--yolo`. |
| `--ask-for-approval never` | toujours | Non-interactivité ; combiné au sandbox, ce qui n'est pas permis échoue au lieu de solliciter. |
| `--ephemeral` | défaut (levé si reprise native activée) | Pas de persistance de session — cloisonnement EXG-ROL-03. |
| `codex exec resume <SESSION_ID>` | optionnel, si `native_resume = true` | Optimisation de reprise (EXG-QUO-02). |
| `--cd <worktree>` | toujours | Périmètre de travail. |
| `-c key=value` | surcharges ponctuelles (désactivation web search, timeouts) | Ajustements sans toucher au fichier de config du poste. |
| `--ignore-user-config` | option de durcissement | Reproductibilité : ignore le `config.toml` personnel du poste (l'authentification reste lue dans `CODEX_HOME`). Activable quand le poste a une config utilisateur non maîtrisée. |
| execpolicy (`.rules`) | règles par rôle générées depuis `policies.toml` | Allowlist de commandes fine (ex. interdire `git push` au TESTER) ; vérifiable hors-run avec `codex execpolicy check`. |
| `AGENTS.md` | **non utilisé** (aucun déployé dans les worktrees) | Le contexte vient du prompt AI-Forge ; un AGENTS.md présent dans un dépôt cible est du contenu de dépôt = données (EXG-SEC-06). L'adaptateur vérifie l'absence d'instructions parasites via le scan standard. |

### 3.3 Vérifications et signaux

- **Health-check** : `codex --version`, `codex login status` (code 0 = authentifié), `codex debug models` pour vérifier la disponibilité de `gpt-5.5`, `codex execpolicy check` sur les règles déployées.
- **Capacités déclarées** : `non_interactive: true`, `json_output: true` (JSONL), `json_schema_output: true`, `model_pinning: true`, `supports_no_attribution: false` → **réécriture des commits par l'orchestrateur** (EXG-INV-03) ; `native_resume: true` ; `native_sandbox: true`.
- **Détection d'épuisement** : limites 5 h/hebdomadaires des abonnements ChatGPT — sortie non-zéro + motifs de limite d'usage dans le flux JSONL/stderr ; patterns calibrés en v0.1.1 dans `providers.toml`.
- **Points d'attention** : les flags globaux se placent **après** le sous-commande (`codex exec --model ...`) ; `--skip-git-repo-check` reste à `false` (les worktrees sont des dépôts Git valides — un échec ici signale un worktree corrompu, information utile).

## 4. Cursor CLI (`cursor-agent`) — modèle imposé : Auto

### 4.1 Invocation type

```bash
# Rôle DEV
cursor-agent -p "$(cat prompt_rendu.txt)" \
  --model auto \
  --output-format json \
  --force \
  # permissions résolues par <worktree>/.cursor/cli.json déployé par AI-Forge

# Rôle TESTER : cli.json restreint (Shell tests/lint uniquement, Write refusé)
# Rôle REVIEWER : --mode=ask (lecture seule, ni édition ni shell)
```

### 4.2 Paramètres retenus et justification

| Paramètre | Valeur AI-Forge | Exigence servie |
|---|---|---|
| `-p` / `--print` | toujours | Mode headless (également inféré en stdout non-TTY). |
| `--output-format json` | `json` (résultat agrégé final) ou `stream-json` (NDJSON temps réel, défaut du print) | EXG-CON ; l'événement terminal `result` porte `is_error`, `result`, `session_id`, durées. **Attention : en échec, aucun JSON bien formé n'est garanti — l'adaptateur traite un flux tronqué + code non-zéro comme ERROR.** |
| `--model auto` | `auto` (configurable) | EXG-ROL-01 — « Auto » est ici la sélection Cursor, conformément à l'arbitrage produit. |
| `--mode=ask` | REVIEWER | Mode nativement **lecture seule** (ni édition ni shell) — traduction directe de la politique REVIEWER. `--mode=plan` disponible pour d'éventuels rôles d'analyse. |
| `--force` | DEV/TESTER | Exécute sans invite d'approbation **dans les limites de `permissions.allow/deny`** — non-interactivité sans permissivité. |
| `<worktree>/.cursor/cli.json` | généré par l'adaptateur depuis `policies.toml`, par rôle | Permissions natives : `Shell(cmd)`, `Read(glob)`, `Write(glob)`, domaines web. Exemple DEV : `{"permissions": {"allow": ["Shell(git)", "Shell(pytest)", "Shell(uv)", "Write(./**)", "Read(./**)"], "deny": ["Shell(rm)", "Read(.env*)", "Read(~/.ssh/**)"]}}`. Le fichier est écrit dans le worktree avant l'invocation et supprimé après. |
| Config globale `~/.cursor/cli-config.json` | non modifiée par AI-Forge | Les règles par-projet (cli.json) suffisent et restent sous contrôle du run. |
| Sessions / reprise | `--resume <session_id>` si `native_resume = true` (le `session_id` est présent dans chaque événement JSON) | Optimisation EXG-QUO-02. |
| Règles / AGENTS.md Cursor | non utilisés | Même politique que Codex : contexte = prompt AI-Forge uniquement. |

### 4.3 Vérifications et signaux

- **Health-check** : `cursor-agent --version`, statut d'authentification (login navigateur effectué sur le poste ; `CURSOR_API_KEY` non utilisé — contrainte « pas d'API » du CDC, l'authentification par abonnement du poste fait foi), invocation d'essai minimale.
- **Capacités déclarées** : `non_interactive: true`, `json_output: true`, `json_schema_output: false` → **contrat extrait par délimiteurs** du champ `result` (EXG-CAP-02, EXG-CON-02) ; `model_pinning: true` (au sens « Auto imposé ») ; `supports_no_attribution: false` → réécriture des commits ; `native_resume: true` ; `native_sandbox: false` → la protection repose sur `cli.json` + diff-guard + (option) conteneur EXG-SEC-04.
- **Détection d'épuisement** : quota fixe de l'abonnement — sortie non-zéro + motifs de limite dans stderr ; particularité : pas de fenêtre de recharge horaire → cooldown long par défaut (configurable, ex. jusqu'au cycle de facturation) dans `providers.toml`.
- **Points d'attention** : le format `stream-json` est le défaut en print — forcer explicitement `--output-format json` si c'est le mode agrégé qui est attendu ; les événements `thinking` sont supprimés en print (aucune fuite de raisonnement dans les transcripts).

## 5. Correspondance rôles × providers (synthèse)

| Rôle | Claude Code | Codex | Cursor |
|---|---|---|---|
| **DEV** | `-p --permission-mode dontAsk` + allowedTools écriture/git/tests | `exec --sandbox workspace-write --ask-for-approval never` + execpolicy DEV | `-p --force` + cli.json DEV |
| **TESTER** | `-p --permission-mode dontAsk` + tools exécution sans Edit/Write/push | `exec --sandbox workspace-write` + execpolicy sans push | `-p --force` + cli.json TESTER (Write refusé) |
| **REVIEWER** | `-p --tools "Read"` + `--json-schema` RoleVerdict | `exec --sandbox read-only` + `--output-schema` | `-p --mode=ask` |
| **Contrat de sortie** | `--json-schema` (natif) | `--output-schema` (natif) | délimiteurs dans `result` |
| **Non-attribution** | setting `attribution` vide (natif) | réécriture des commits | réécriture des commits |
| **Cloisonnement** | `--bare --no-session-persistence --strict-mcp-config` | `--ephemeral` (+ `--ignore-user-config` en durcissement) | sessions non réutilisées + cli.json par run |

## 6. Squelette `providers.toml`

```toml
[claude]
bin = "claude"
model = "opus-4.8"
invocation = ["-p", "--output-format", "json", "--bare", "--strict-mcp-config",
              "--no-session-persistence", "--exclude-dynamic-system-prompt-sections"]
settings_inline = '{"attribution": {"commit": "", "pr": ""}}'
max_turns = { dev = 80, tester = 40, reviewer = 20 }
capabilities = { non_interactive = true, json_output = true, json_schema_output = true,
                 model_pinning = true, supports_no_attribution = true,
                 native_resume = true, native_sandbox = false }
exhausted_patterns = []        # calibrés en v0.1.1 sur messages réels
cooldown = { kind = "window", hours = 5, weekly = true }
max_concurrency = 2

[codex]
bin = "codex"
model = "gpt-5.5"
subcommand = "exec"
invocation = ["--json", "--ask-for-approval", "never", "--ephemeral"]
sandbox = { dev = "workspace-write", tester = "workspace-write", reviewer = "read-only" }
profiles = { dev = "ai-forge-dev", tester = "ai-forge-tester", reviewer = "ai-forge-reviewer" }
capabilities = { non_interactive = true, json_output = true, json_schema_output = true,
                 model_pinning = true, supports_no_attribution = false,
                 native_resume = true, native_sandbox = true }
exhausted_patterns = []
cooldown = { kind = "window", hours = 5, weekly = true }
max_concurrency = 2

[cursor]
bin = "cursor-agent"
model = "auto"
invocation = ["-p", "--output-format", "json", "--force"]
reviewer_mode = "ask"
cli_json_per_role = true       # .cursor/cli.json généré dans le worktree
capabilities = { non_interactive = true, json_output = true, json_schema_output = false,
                 model_pinning = true, supports_no_attribution = false,
                 native_resume = true, native_sandbox = false }
exhausted_patterns = []
cooldown = { kind = "fixed" }  # quota fixe : cooldown long configurable
max_concurrency = 2
```

## 7. Paramétrage pour le développement d'AI-Forge lui-même (v0.0.1)

AI-Forge est développé **par** les trois CLI avant de savoir les orchestrer. Cette annexe sert donc deux usages distincts, avec une inversion assumée sur les mécanismes de contexte implicite :

| | Développement d'AI-Forge (sessions pilotées par l'humain) | Runtime AI-Forge (invocations orchestrées) |
|---|---|---|
| CLAUDE.md / AGENTS.md / règles | **Utilisés délibérément** : ce sont les vecteurs des conventions du projet. | **Neutralisés** (`--bare`, non-déploiement) : le contexte vient du module `context/`. |
| Sessions | Interactives, persistantes, reprises librement. | Headless, éphémères, cloisonnées par rôle. |
| Permissions | Confortables mais bornées (deny sur destructif et secrets). | Strictes par rôle (§5). |
| Non-attribution (INV-006) | **Identique : active dès le premier commit.** | Identique. |
| Pinning de modèle | Identique (Opus 4.8 / GPT-5.5 / Auto). | Identique. |

### 7.1 Fichiers committés à la racine du dépôt `ai-forge` (v0.0.1)

- **`CLAUDE.md`** (Claude Code) et **`AGENTS.md`** (Codex ; également lu par Cursor) : contenu identique sur le fond — normes de développement du CDC §4 (POO, une classe par fichier, docstrings reStructuredText, `mypy --strict`, couverture ≥ 95 %, black/ruff/bandit, Conventional Commits), definition of done d'un BL, renvoi vers le CDC et les annexes, rappel explicite d'INV-006 et de la hiérarchie des instructions (EXG-SEC-06).
- **`.claude/settings.json`** (versionné) : `{"attribution": {"commit": "", "pr": ""}}`, `permissions.allow` (uv, pytest, ruff, mypy, black, bandit, git hors push forcé), `permissions.deny` (`rm -rf`, lecture `~/.ssh` et magasins de secrets), modèle par défaut `opus-4.8`.
- **`.cursor/cli.json`** (versionné) : `permissions.allow/deny` équivalentes.
- **Profils Codex** : `~/.codex/config.toml` étant hors dépôt, un fichier `docs/dev-setup/codex-profiles.toml` versionné fournit les profils à installer (`ai-forge-dev`, modèle `gpt-5.5`, sandbox `workspace-write`), et `forge doctor` (à partir de la v0.1.x) vérifie leur présence.
- **`docs/dev-setup/README.md`** : procédure d'installation des trois CLI, vérification des modèles, activation des configurations ci-dessus — c'est la première chose qu'un contributeur (humain ou IA) exécute.

### 7.2 Processus de développement assisté (avant self-hosting)

Jusqu'à ce qu'AI-Forge sache exécuter ses propres BL (self-hosting visé à partir de la v0.2.0), le cycle du CDC est appliqué **manuellement** : l'humain lance une session CLI par BL (dans un worktree dédié s'il y a parallélisme), l'IA développe selon les conventions des fichiers ci-dessus, la PR est contrôlée par la CI, la revue est confiée à une session d'une **autre** CLI, l'humain merge. Chaque BL d'AI-Forge est ainsi développé dans les conditions que l'orchestrateur automatisera ensuite — le paramétrage est éprouvé en conditions réelles avant d'être codé dans les adaptateurs, et les patterns d'épuisement (`exhausted_patterns`) sont capturés au fil de ces sessions.

## 8. Ce que cette annexe impose au CDC

1. La capacité `json_schema_output` est ajoutée à `ProviderCapabilities` (Claude et Codex valident nativement les contrats — la relance EXG-CON-02 devient un cas rare réservé à Cursor).
2. La capacité `native_sandbox` est ajoutée : Codex fournit un sandboxing OS natif qui complète EXG-SEC-01/04 sans conteneur.
3. La non-attribution (INV-006) a une implémentation native chez Claude Code (setting `attribution`) et repose sur la réécriture des commits chez Codex et Cursor — la réécriture reste donc obligatoire dans le tronc commun de l'orchestrateur.
4. Les mécanismes de contexte implicite des trois CLI (CLAUDE.md, AGENTS.md, règles Cursor, MCP, hooks, skills, mémoire automatique) sont **désactivés ou ignorés** : le module `context/` d'AI-Forge est l'unique source de contexte (EXG-CTX, EXG-SEC-06).
5. La calibration des `exhausted_patterns` est un livrable explicite de la v0.1.1 (tests en conditions réelles d'épuisement ou simulation par messages capturés).
